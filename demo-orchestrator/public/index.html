RGB asset transfers no one can read or block

To complete an RGB transfer, the sender must hand the receiver a consignment — and today that hand-off runs through centralized proxy servers that can read it and censor it. Parcel21 moves it onto Nostr, encrypted end-to-end: the relay sees only ciphertext, the receiver is pushed the data in real time, and anyone can run a relay. This page runs a real one and hands you the receipts.

The relay is just an encrypted mailbox — it never learns the asset, the amount, or who is transacting. That's the point.

    Drop-in for the RGB proxy protocol

    Parcel21 mirrors RGB HTTP JSON-RPC so wallets can adopt it as an alternative transport. Each method maps to a Nostr mechanism:

    consignment.postSender gift-wraps the consignment to the receiver (kind 1059)
    consignment.getReceiver subscribes for gift wraps addressed to it and unwraps
    ack.postReceiver gift-wraps an ACK to the sender's per-transfer reply key
    ack.getSender is pushed the ACK — no polling

    Encryption & transport

    Payloads are encrypted with NIP-44 and wrapped with NIP-59 gift wrap, addressed to an ephemeral key — so the relay sees an opaque blob, not the contents or the real parties. Small consignments ride inline; larger ones go to encrypted Blossom storage referenced inside the wrap.

    Security model

    Authenticated unwrap (a normative MUST). A naive gift-wrap unwrap does two decryptions and no authentication — so a forged wrap could fake an ACK and trigger a Bitcoin broadcast. Parcel21 verifies the seal signature, binds the seal's key to the rumor author, and recomputes the rumor id on every unwrap, and only broadcasts the witness tx after a verified ACK. You can watch that ordering in the live run above.
