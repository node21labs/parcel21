# Parcel21 Architecture

> Status: **draft**. Provisional details (event-kind numbers, exact crate/library versions) are marked
> `‹provisional›` and finalized as the NIP and reference implementations stabilize.

Parcel21 replaces the centralized RGB consignment proxy with a Nostr-based coordination layer. The
design is intentionally faithful to the existing [RGB HTTP JSON-RPC v0.2](https://github.com/RGB-Tools/rgb-http-json-rpc)
protocol so that wallets can adopt it as an alternative transport without changing the surrounding RGB
transfer workflow.

## 1. Background: the RGB transfer it must serve

From the [RGB sandbox](https://github.com/RGB-Tools/rgb-sandbox) flow, an RGB transfer proceeds:

1. **Receiver** generates an invoice containing a **blinded UTXO** (the `recipient_id`) and one or more
   **transport endpoints** telling the payer where to deliver the consignment.
2. **Sender (payer)** runs `rgb transfer`, producing a **consignment file** and a PSBT committing to it.
   The witness transaction is **not yet broadcast**.
3. The consignment must reach the receiver. *This is the hop Parcel21 decentralizes.*
4. **Receiver** validates the consignment and signals **ACK** (valid) or **NACK** (invalid).
5. On ACK, the **sender** signs, finalizes and **broadcasts** the Bitcoin transaction.

The `recipient_id` (blinded UTXO) is the rendezvous key: it is generated by the receiver, embedded in
the invoice, and therefore **known to both parties** — but it reveals nothing about the receiver's
on-chain identity (the outpoint is blinded).

## 2. The three layers

```
┌─────────────────────────────────────────────────────────────────────┐
│  CLIENT LAYER   (client/)                                             │
│  Wallet / web app. Builds & validates RGB consignments, drives the    │
│  send → receive → validate → ACK/NACK flow, manages Nostr keys.       │
└───────────────┬─────────────────────────────────────┬────────────────┘
                │ encrypted, gift-wrapped events        │ encrypted blob refs
                ▼                                       ▼
┌───────────────────────────────────┐   ┌──────────────────────────────┐
│  COORDINATION LAYER (Nostr)       │   │  STORAGE LAYER                │
│  relay/  — purpose-built relay.   │   │  Inline (small) OR encrypted  │
│  Carries NIP-59 gift wraps of     │   │  Blossom blob (large). Server │
│  consignment / ack / media events.│   │  sees only ciphertext.        │
│  Cannot read contents or parties. │   │                               │
└───────────────────────────────────┘   └──────────────────────────────┘
```

### Coordination layer (Nostr)
Carries **encrypted metadata** between sender and receiver: pointers to consignment data, the
`recipient_id` correlation, and ACK/NACK responses. Every payload is encrypted with **NIP-44** and
wrapped with **NIP-59 gift wrap**, so a relay operator sees only an opaque blob addressed to an
**ephemeral** public key — not the contents, and not the real sender or receiver. Receivers hold an open
subscription and are notified in real time.

### Storage layer
Storage-agnostic. Small consignments are embedded directly in the (encrypted) event. Larger ones are
encrypted, uploaded to a [Blossom](https://github.com/hzrd149/blossom) server as a **ciphertext blob**,
and referenced by `{server, sha256, key}` *inside* the gift wrap — so neither the relay nor the Blossom
server ever sees plaintext. Threshold for inline vs external: **~32–40 KB raw inline**, else Blossom — base64
inflation plus NIP-44's double wrap (65535-byte plaintext limit per layer) cap the inline path (NIP §6).

### Client layer
Ties coordination + storage together. Mirrors the proxy workflow (post → retrieve → ACK/NACK) over
decentralized infrastructure. The reference client is a [TanStack Start](https://tanstack.com/start) app.

## 3. Method mapping (proxy → Nostr)

The proxy exposes seven JSON-RPC methods. Parcel21 maps each to a Nostr mechanism while preserving the
proxy's **immutability** guarantees (a posted consignment and a set ACK cannot change).

| Proxy method | Params | Parcel21 mechanism |
|---|---|---|
| `server.info` | — | Relay [NIP-11](https://github.com/nostr-protocol/nips/blob/master/11.md) info document advertises Parcel21 support + limits |
| `consignment.post` | `recipient_id`, `txid`, `vout?` + file | Payer gift-wraps a **consignment** rumor (kind `1517`, provisional) to the payee; carries `recipient_id`, `txid`, and the consignment (inline or Blossom ref) |
| `consignment.get` | `recipient_id` | Payee subscribes for gift wraps (kind `1059`) p-tagged to it, unwraps, matches on `recipient_id` |
| `ack.post` | `recipient_id`, `ack:bool` | Payee gift-wraps an **ack** rumor (kind `1518`, provisional) to the payer's per-transfer reply key; `ack` true=ACK / false=NACK |
| `ack.get` | `recipient_id` | Payer subscribes for the ack gift wrap — **push, not poll** |
| `media.post` | `attachment_id` + file | Same as `consignment.post`, content addressed by `attachment_id`; large media → Blossom |
| `media.get` | `attachment_id` | Same as `consignment.get` |

## 4. The ACK return-path problem

In the proxy model the payer is **anonymous** — it posts a consignment and later *polls* `ack.get`. Over
Nostr there is no server to poll: the payee must actively **deliver** the ACK back to the payer. Parcel21
solves this without de-anonymizing the payer:

1. The payer generates a **per-transfer ephemeral Nostr keypair** `K_reply`.
2. The payer includes `pubkey(K_reply)` *inside* the encrypted consignment rumor as a reply-to.
3. The payee gift-wraps its ACK/NACK addressed to `pubkey(K_reply)`.
4. The payer keeps a subscription open for gift wraps to `pubkey(K_reply)` and receives the ACK.

Because `K_reply` is fresh per transfer and only ever appears inside encrypted payloads (never linked to
the payer's long-term identity), the relay learns nothing. This is validated against NIP-59 semantics in
the design-verification pass; see [`docs/roadmap.md`](roadmap.md).

> Open question carried into the NIP: payer-offline delivery. Unlike a polled server, push delivery
> assumes the payer can reconnect and re-query the relay for stored gift wraps. Handled by using
> **stored** (not ephemeral) event kinds + relay retention, plus optional [NIP-40](https://github.com/nostr-protocol/nips/blob/master/40.md) expiry.

## 5. Invoice transport endpoint

RGB invoices already carry transport endpoints (e.g. `rgbhttpjsonrpc:https://proxy.example/json-rpc`).
Parcel21 introduces a **Nostr transport type** so a payee can advertise *where* and *to whom* to deliver
a consignment — encoding the relay URL(s) and the payee's Nostr pubkey. Wire syntax (NIP §5, provisional):
`rgbnostr:<payee-npub>?relay=wss://relay.parcel21.example`, carried in the RGB invoice's `endpoints=` list
alongside existing transports like `rgbhttpjsonrpc:`.

## 6. What a relay still sees (threat model, in brief)

| Observer | Sees | Does NOT see |
|---|---|---|
| Relay | Opaque gift-wrap events to **ephemeral** keys, sizes, timing | Plaintext, real sender/receiver, `recipient_id`, `txid` |
| Blossom server | Ciphertext blob + downloader IP | Plaintext consignment, the parties, the `recipient_id` |
| Invoice observer | `recipient_id` (blinded UTXO), payee npub, relay URLs | Consignment contents, whether/when delivered |

Full threat model and the adversarial review of the mapping land in the NIP and `docs/`.

## 7. Component map

- **`spec/`** — the NIP. Normative event kinds, tag structures, encryption, flow, error semantics.
- **`relay/`** — Nostr relay (TypeScript/Bun + Postgres, Drizzle ORM), Docker-packaged with an admin UI.
  Implements NIP-01/09/11/40/42/70; an opt-in Parcel21 write-policy (`RELAY_KIND_ALLOWLIST`) restricts it to
  gift-wrap `kind 1059`, and `MAX_MESSAGE_BYTES` bounds event size for consignment-sized events.
- **`client/`** — TanStack Start reference client. Reads invoices, gift-wraps consignments, subscribes
  for delivery + ACK, drives validate/ACK/NACK. Nostr lib: [`nostr-tools`](https://github.com/nbd-wtf/nostr-tools) `2.23.x`
  (TanStack Start v1.0 RC, pin `^1.168`).
